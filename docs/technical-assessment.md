# 技术评估报告 — SDK Agent Readiness Eval（AX eval）

**写给：** 项目负责人（数据科学背景，熟悉 AI 应用，对 DevOps / AI infra 了解较浅）
**目的：** 一份"深入浅出的 executive summary"——既讲清楚商业价值，也把技术架构和每个陌生概念讲透，作为后续速成班的教材。
**评估日期：** 2026-06-01
**评估对象：** 当前 repo 的全部文档（5 份策略/规格）+ 已落地的 TypeScript 代码骨架 + 测试套件。

> 一句话总结：**这是一个测"AI agent 到底能不能真正用你的产品完成任务"的工具。** 现在市面上所有人都在测"你有没有把接口暴露出来"（静态体检），但没有人测"暴露了之后 agent 用起来到底成不成功"（行为验证）。这个项目要占的就是后面那个空位。代码层面，一个**可以无密钥端到端跑通的骨架**已经建好了，真正接真实 agent 的部分还没动工。

---

## 第一部分：这是什么，以及为什么有潜力（商业视角）

### 1.1 用大白话讲清楚要做的事

想象你是 Asana（一个项目管理 SaaS）的开发者关系团队。过去你的"用户"是人类程序员——他们读你的文档，照着写代码，集成你的产品。现在多了一类新用户：**AI agent**（比如 Claude Code、Cursor、Codex 这种能自己读文档、自己写代码、自己调 API 的编程助手）。

问题来了：你的文档对人类写得很清楚，但 AI agent 读了之后，能不能**真的成功地**"在 Marketing 项目里创建一个任务、指派给某人、设个截止日期"？

- 你可能觉得"我有 API、有 OpenAPI 规范、有 MCP server、有 SDK，肯定能用啊"。
- 但**暴露了接口 ≠ agent 真的会用**。文档里少一个例子、某个字段描述有歧义、认证流程写得绕，agent 就会失败。

**这个项目就是去实测这件事。** 它让真实的 agent 去跑真实的任务，然后告诉你：
1. 成功率多少（在 Claude Code 里 8 个任务成 5 个，在某个弱一点的 agent 里只成 3 个）；
2. 具体卡在哪一步（认证？找错端点？字段理解错？）；
3. 怎么改文档/规范能修好。

并且是**持续地**做——每次你改了文档或 API，就重新跑一遍，像回归测试一样。

### 1.2 一个绝佳的类比：这是"Agent 体验"的集成测试

文档里反复用一个类比，非常精准，请记住它，因为它是整个产品的灵魂：

| 软件测试世界 | 这个项目对应的角色 | 它回答的问题 |
|---|---|---|
| **Lint（代码风格检查）** | 静态体检（别人在做） | "你的代码格式规范吗？"→ "你有没有 llms.txt / OpenAPI？" |
| **Code Review（人工评审）** | 编辑式榜单（别人在做） | "第三方觉得你写得好不好？" |
| **集成测试（真跑一遍）** | **本项目** | "把所有东西拼起来，真的能跑通吗？" |

**Lint 和 Review 都不真正运行程序；集成测试才真正运行。** 这个项目就是 Agent 体验领域里那个"真正运行一遍"的环节，而这个环节目前是空白。

### 1.3 三层市场地图（为什么这个空位值钱）

文档把"SaaS 在 agent 时代必须赢的战场"分成三层：

1. **被发现 / 被选中**（Discovery）——agent 会不会选你？这是 AEO/GEO（答案引擎优化）的地盘，已经很卷。
2. **暴露可操作接口**（Interface Exposure）——你有没有 MCP / CLI / OpenAPI / SDK / 文档？这是 **AX（Agent Experience，Agent 体验）/ agent-readiness** 的地盘，已经有玩家（Cloudflare 的 Agent Readiness 评分等）。
3. **操作能否成功**（Operational Success）——agent 真的去用了，任务到底完成没完成？**这一层是空的。这就是本项目的位置。**

核心洞察一句话：**"暴露能力 ≠ 能力可用"。** 一家公司可以同时上线 MCP + CLI + 文档，agent 端到端跑还是失败。本项目就是给这个落差当"温度计"。

### 1.4 这个洞察不是凭空想的——有学术证据撑腰

2026 年 EASE 会议有一篇论文（Lima 等人，*Making OpenAPI Documentation Agent-Ready*，项目里 hermes harness 的命名就来自它）：

- 用**原始未优化**的 OpenAPI 文档，agent 自动化任务约 **70% 在规划阶段就失败**；
- 把文档质量改好后（API 本身没变），成功率跳到 **90%**。
- 结论原话：**"结构上正确并不等于 agent-ready；瓶颈不是 AI 能力，而是文档/资料的成熟度。"**

这正是本项目要测、并要帮客户修的那个落差——而且是同行评审论文给出的硬证据，是销售时最有力的弹药。

### 1.5 竞品分析：为什么说这个空位真的空着

项目做了相当扎实的竞品调研，结论是市面上的东西都落在两个桶里，**都不是这个形状**：

- **桶 A：静态体检（= lint）。** 代表是 Cloudflare 的 "Agent Readiness" 评分（isitagentready.com）——像网页性能工具 Lighthouse 那样，给你的站点在"可发现性、内容、机器人访问、能力暴露"四个维度打分。**它从不真正跑任务。** 这是别人的强项，我们不去抢。
- **桶 B：编辑式榜单（= 评测文章）。** 代表是 Tech Stackups 的 "AX Benchmark"——一篇方法跟我们很像的文章（让 Claude Code 从零集成某服务，测耗时/人工介入次数），但**承诺的持续榜单从没真正上线**，被判定为"AI 内容站发的一次性文章，不是产品"。

> **关键差异：** 别人是"一次性地、从外部、测一个 agent 然后排名"。本项目是"让 SaaS 自己、在 CI 里、持续地、跨多个 agent"测自己的产品。**Benchmark ≠ Platform；别人测你 ≠ 你测自己。** 这个连 Cloudflare 都没在卖。

一个值得警惕的信号：**Anthropic 在 2026-05 收购了 Stainless**（一家"把 API 规范自动生成 SDK/CLI/MCP"的公司，>$3 亿）。这说明"让产品 agent 可操作"这件事战略上很热——但大厂下注的是**生成/暴露**那一层（第 2 层），不是**验证**这一层（第 3 层）。我们的空位还在，但要"盯着大厂"，因为拥有生成层的人天然有可能往验证层延伸。

### 1.6 商业模式：开源是漏斗，托管是生意

这是让它"成为一门生意而不只是开源仓库"的关键决策：

- **开源（免费）：** 一个本地跑的 skill / CLI 工具 + 四个数据格式标准（schema）+ 单次单 agent 的本地运行。
  - 目的：拿 GitHub star、自下而上铺开、**让我们的 schema 成为行业事实标准**（标准被采纳 = 真正的护城河）+ 让社区/厂商贡献"target pack"（每个 SaaS 的测试包），解决"难道每个 SaaS 都要我们手工搭"的扩展性问题。
- **托管（付费）：** 跨多 agent 的大规模矩阵 + CI 集成 + 历史记录与回归告警（"这个 PR 把分数从 84 拉到了 61"）+ 深度诊断与修复建议 + 行业百分位对标。**这些只有作为托管服务才成立**，是真正的生意。

商业上还有一个聪明的防御点：**中立性**。我们跨 Claude/GPT/Gemini/开源 agent 测；单个大厂（尤其是收了 Stainless 的 Anthropic）不会中立地这么做。

### 1.7 一句话的销售钩子（值得记住）

> "你已经有 code-sample CI 了——它告诉你你的代码**能跑**。我们告诉你 agent 能不能真正**用懂**你的文档。这俩不是一回事。"

## 第二部分：技术架构详解（把每个陌生概念讲透）

这一部分是给你的"速成班正餐"。我会先解释会反复出现的陌生名词，再讲系统怎么搭起来。

### 2.0 先扫清术语（DevOps / AI infra 黑话）

读这个 repo 你会撞到一堆词，先一次性讲清楚：

- **Agent（智能体）：** 不只是调一次大模型，而是一个"循环"：模型读任务 → 决定调用某个工具（比如发 HTTP 请求、读文件）→ 看结果 → 再决定下一步……直到完成。Claude Code、Cursor、Codex 都是 agent。
- **Harness（运行支架）：** agent 的"外壳/脚手架"——具体由哪个模型 + 哪些工具 + 怎么检索文档 + 多大上下文窗口 + 怎么循环组成。**同一份文档，在一个 harness 里能成功、换个 harness 就失败**，所以才要"跨 harness"测试。可以理解为：模型是发动机，harness 是整辆车。
- **MCP（Model Context Protocol）：** Anthropic 主推的一个开放协议，让 agent 用统一方式连接外部工具/数据源。一个 SaaS 提供"MCP server"，agent 就能即插即用地调用它的能力。你可以把它类比成"给 agent 用的 USB 接口标准"。
- **OpenAPI（规范）：** 一份机器可读的文件（JSON/YAML），用结构化方式描述一个 REST API 的所有端点、参数、返回值。人和 agent 都能据此知道"这个 API 怎么调"。
- **llms.txt / AGENTS.md：** 新兴的约定文件。`llms.txt` 放在网站根目录，专门给大模型读的站点说明（类比给搜索引擎的 `robots.txt`，但内容是"喂给 LLM 的导航"）。`AGENTS.md` 是给编程 agent 的项目说明。它们都属于"把内容做得机器可读"的静态准备。
- **CLI（命令行工具）：** 在终端里敲命令运行的程序（区别于图形界面）。本项目本身是个 CLI，它驱动的 agent（Claude Code、Codex）也都能以 CLI 方式"无界面（headless）"运行。
- **Headless（无界面运行）：** 让一个本来交互式的工具，用一条命令跑完、输出结果，不需要人在中间点。比如 `claude -p "做这个任务" --output-format json`。这是能把 agent 塞进自动化流水线的前提。
- **BYOK（Bring Your Own Key，自带密钥）：** 用户提供自己的 API key，agent 的推理花费记在用户账上，我们不替他付费、也不碰他的密钥。
- **CI（Continuous Integration，持续集成）：** 每次代码/文档改动，自动跑一遍测试的流水线（如 GitHub Actions）。"在 CI 里持续跑"= 每次改文档自动重测 agent 成功率。
- **Oracle（判定器）：** 测试领域的术语，指"判断一次运行到底算不算成功"的那个权威。**这是本产品的命门**（见 2.3）。
- **Sandbox（沙箱）：** 一个隔离的、可随便折腾的测试环境（比如一个专门的 Asana 测试 workspace），agent 在里面真的去创建/删除东西，不会污染生产数据。
- **AEO / GEO / AX：** AEO=Answer/Agentic Engine Optimization（让 agent 发现/选中你，偏营销/SEO）；GEO=Generative Engine Optimization（更大的伞）；**AX=Agent Experience（Agent 体验，本项目认领的伞）**——agent 如何发现、理解、并**操作**你的产品。

### 2.1 两层评测 + 一个落差：整个产品的骨架

系统对一个目标产品做**两层评测**，最后汇成一个报告。这是整个产品的核心结构图（已经体现在代码里）：

```
   行为层（Behavioral，核心/护城河）         静态层（Static，免费钩子）
   ──────────────────────────────         ──────────────────────────
   真实任务 × 真实 agent harness            体检公开接口：
   ↓ agent 真的去操作产品                    llms.txt / AGENTS.md /
   ↓ oracle 查真实状态，判定成功/失败          OpenAPI / MCP / SDK /
   ↓                                        robots+sitemap / OAuth
   每个 agent 的通过率 + 跨 agent 差异        ↓
                                            0–100 的"agent-readiness"分
              │                                      │
              └──────────────┬───────────────────────┘
                             ▼
                   THE GAP（落差，卖点所在）
            "静态分 92（绿），但最强的 agent 只成功 3/10"
            = 暴露 ≠ 可用 的实证版
```

- **静态层（`audit` 命令）：** 只读你公开的 URL，不需要密钥、不需要 agent。问的是"管道铺好了吗"。这是**免费的引流钩子**——任何人都能克隆一个静态体检，所以它不是护城河，是诱饵。
- **行为层（`run` 命令的矩阵）：** 真的让 agent 去跑任务。问的是"agent 真能干成活吗"。**这是核心、护城河、付费项**——别人都不卖这个。
- **落差（the gap）：** 把两个分数并排放。**"静态高 × 行为低"才是独家卖点**——这是产品核心论点（暴露≠可用）的可量化证据。

### 2.2 四个核心数据格式（schema）——真正的"标准/护城河"

项目反复强调：**真正要推广成行业标准的不是代码，而是四个数据格式。** 谁的格式被大家采用，谁就拥有标准。这四个是：

1. **Task（任务）** — 一个目标级的活 + 它的成功判定。
   ```
   id: asana-create-task
   prompt: "在 Marketing 项目里创建一个名为 'Draft launch plan' 的任务。"
   oracles: [ {type: exists, path: task.gid}, {type: equals, path: task.name, expected: "Draft launch plan"} ]
   ```
2. **Target Pack（目标包）** — 让"一个 SaaS 可被测"的全部打包：它暴露了哪些接口、用什么认证、setup/reset 钩子、还有任务集。**这是回答"难道每个 SaaS 都要中心化地手搭沙箱吗"的关键抽象——不需要，格式开放，谁都能填，社区众包。**
3. **Harness Adapter（运行支架适配器）** — 一层薄薄的包装：把真实 agent CLI 以无界面方式启动 → 注入 BYOK 密钥 → 把它原生的输出"归一化"成统一结果。
4. **RunResult（运行结果）** — 不管哪个 agent 跑出来的，都长一个样，这样才能跨 agent 比较。

> 工程上这四个 schema 现在用 **Zod**（一个 TypeScript 的运行时类型校验库）定义在 `src/schemas.ts`，加载 target pack 用 **YAML** 格式。这是已经落地的部分。

### 2.3 Oracle（判定器）——整个产品的命门

文档原话：**"成功判定器就是全部赌注（the whole ballgame）。"** 每一个分数的可信度，最终都归结到一件事：**我们凭什么说一个任务成功了？**

有两种判定方式：

- **程序化 oracle（强烈首选）：** 任务跑完后，**用代码去查真实服务**，断言世界真的按要求变了——一个确定性的 true/false。**我们不看 agent 嘴上说"我做完了"，我们查现实。** 这能免疫"agent 自信地撒谎说做完了但其实没做"。
- **LLM 裁判（仅作兜底）：** 把运行记录丢给一个模型问"它成功了吗"。主观、可被自信的 agent 糊弄、可复现性差。

**这就是为什么第一个目标选 Asana** 而不是"发邮件"：Asana 几乎每个操作（建任务、指派、设字段、移动分区）都会留下**可以通过 API 查回来的状态**，所以大多数任务都能拿到干净的程序化判定，而不用退化成 LLM 裁判。（对比"发邮件"：验证成功得真去翻收件箱，很难。）选 Asana 悄悄地把产品最难的部分给降险了。

> **数据科学家的类比：** oracle 就是你的 ground-truth label。如果 label 本身不可信（噪声大、主观），那再漂亮的"准确率"也没意义。程序化 oracle = 干净的客观标签；LLM 裁判 = 用另一个模型当标注员，有偏差且不可复现。整个产品的可信度 = label 质量。这点你应该比一般工程师更有共鸣。

代码里（`src/oracles.ts`）现在实现了三种程序化判定，作用在一个"世界状态"对象上，用点号路径寻址（如 `task.gid`、`task.due_on`）：
- `exists`：某个路径存在吗？（任务被创建了吗）
- `equals`：值精确相等吗？（截止日期对吗）
- `contains`：数组里有这个元素 / 字符串里含这个子串吗？（评论文本在不在 stories 里）

一个任务的成功 = **它所有 oracle 全过**。

### 2.4 Harness Adapter（适配器）——工程护城河

一个关键决策：**绝不克隆/重写 agent，而是编排真实的 agent。** 因为克隆出来的会和真实产品漂移，毁掉"测的就是用户真正在用的东西"这个可信度根基。

两个旗舰 agent 都已经支持无界面运行：
- **Claude Code：** `claude -p "<任务>" --output-format json --max-turns N`
- **Codex：** `codex exec --sandbox workspace-write --output-schema schema.json "<任务>"`

适配器要做的事很薄：**启动 → 注入密钥 → 把原生输出归一化成统一的 RunResult。** 但"这套适配器 + 归一化 + 版本锁定"合起来**就是护城河**——因为这两个 CLI 几乎每周都更新，维护这层适配很烦、很费劲，是真正的工程壁垒。

**为什么必须"跨 harness"：** "跨多个 agent"只有当这些 agent 在真正影响文档消费方式的维度上不同时，才有意义。所以每个 harness 要用一个"特征画像"描述并刻意张开这个空间：
- **检索策略**（抓 llms.txt？还是 grep？还是把全文塞进上下文？）
- **工具访问**（能不能联网抓取？有没有 MCP 客户端？能不能跑 shell？）
- **上下文窗口大小**
- **模型**（以及推理强度）
- **自主度**（遇事就问人，还是全自动）

计划的 harness 组合：品牌款（Claude Code、Codex，给可信度）+ 几个开源款（Aider、OpenHands、Goose、Cline，便宜、可锁版本、铺满矩阵）+ **一个刻意做弱的"最小 harness"当下限基线**（支撑"在 Claude Code 里 81 分、在最小 harness 里 38 分"这种差异故事）。

### 2.5 完整执行流程（把上面串起来）

一次 `run` 命令发生的事：

```
1. CLI 解析命令和参数 → 加载 .env（密钥）和 target pack（YAML，经 Zod 校验）
2. 行为层：runner 遍历 每个任务 × 每个 harness
      每个 harness 报告一个"世界状态" + 运行轨迹
      → oracle 给世界状态打分 → 每个格子得到一个 RunResult
3. 静态层：auditSite() 并发跑 8 项体检 → 加权 0–100 分
4. 报告：渲染 任务×harness 矩阵 + 各 harness 通过率
      + 静态分 vs 最强真实 agent 通过率的"落差"
      → 存成 JSON
```

### 2.6 北极星 UX：「丢个链接」自动评测 + 人工审核闸门（v1+ 愿景）

这是产品想达到的最终形态（现在还没建，但值得知道，因为它定义了天花板）：

1. 开发者装上 skill，提供两类密钥：**agent/模型密钥**（自己的推理花费）+ **自己产品的沙箱认证**（让 agent 真能去操作产品）。
2. **丢进自己的文档站 URL。**
3. skill 自动读文档、发现 OpenAPI/llms.txt/MCP，**自动合成一套评测任务 + 起草 oracle** + setup/reset。
4. **强制的人工审核闸门（任何东西执行前必过）：** 开发者审阅、修改自动生成的任务和判定器——可以手改，也可以让 AI 改，但**最终必须有人类显式批准**（不能"AI 批准 AI"）。
5. skill 跑 harness 矩阵 → 诊断报告。

为什么这个闸门是关键？它一次干三件事：
- **可信度** — 自动生成的 oracle 是风险最高的东西，一个"自信地错"的 oracle 会给出可信但虚假的分数。人工批准把"自动（有风险）"变成"自动起草 + 人工批准（可信）"。
- **安全** — agent 会对沙箱做真实的**写操作**，审核是开发者确认"它要干什么"以及"确实指向沙箱不是生产"的地方。
- **数据飞轮** — 每次人工修改都是 ground-truth 信号，反过来训练/改进生成器。

配套的 **oracle 分级**（决定什么能自动起草、可信度多高）：
- **T1 — OpenAPI CRUD 往返：** 从规范推导验证（任务说 POST 创建 X → 用 GET 查回来断言字段）。自动、高可信、真程序化。**这是让"丢链接"能产出真实分数的甜点区**，覆盖一大类有 OpenAPI 的对称 CRUD API。
- **T2 — 弱信号：** "调用返回了 2xx / 没报错"，或 LLM 裁判。自动、低可信、**永远打标签**，绝不让弱判定冒充强判定。
- **T3 — 人工策划：** 人手写的（v0 的 Asana 集就是；付费/服务层）。最高可信。

### 3.1 技术栈

- **语言：** TypeScript（已锁定），跑在 Node.js 22+。

### 3.2 ✅ 已经真正建成、能跑的部分

| 模块 | 状态 | 说明 |

### 3.3 ⚠️ 是"桩"（stub）/ 占位，不是真东西

**这是你必须清楚的最重要一点：现在还没有任何真实的 agent 在跑。** 所谓的行为矩阵，跑的是**假 agent**：

- **`mock`（满分假 agent）：** 不调任何真实 API。它直接照着任务的 oracle 反向"伪造"一个正确的世界状态，所以全过。它被自动标记为"合成对照组"，在落差计算里被排除。
- **`mock-weak`：** 同样是假的，但被配置成故意跳过 2 个任务、做错 1 个，制造"真实感"的失败分布（通过率约 62.5%）。
- **`hermes`：** **是个无密钥的桩，不是真的 LLM。** 它继承自 mock，运行时在轨迹开头打一行 `[hermes] NOTE: keyless stub — provider/auth TBD; output is simulated`（明确声明输出是模拟的）。它的真实 provider/认证还没定。

这些假 harness 存在的意义是：**让整条管道在没有密钥、没有真实 agent 的情况下也能端到端跑通和测试**——这是合理的工程做法（先把骨架接通），但要清醒：**目前矩阵里的数字是模拟的，不是真实 agent 跑出来的。**

### 3.4 ❌ 还完全没建（需要密钥 / 后续里程碑 M1+）

- **真实的 agent 适配器：** `claude-code`（`claude -p`）、`codex`（`codex exec`）、以及那个"最小 harness"——**一个都还没写**。代码里留了注册位置（registry），但适配器本身是空的。

- **沙箱的 setup/reset：** `targets/asana/setup.ts`（建沙箱项目）和 `reset.ts`（每次尝试间清理，避免 pass@k 互相污染）——还没写。**reset 对可重复的真实运行是关键**。
- **编辑式对比评测集**（Asana vs Linear vs Monday）——推迟。
- **"丢链接"自动生成 UX + 审核闸门**（第 2.6 节那套）——v1/v2 愿景，未建。

## 第四部分：技术评估——优点、风险、缺口

### 4.1 工程优点

1. **架构分层干净，扩展点清晰。** 适配器是抽象基类，加新 agent 只要继承 + 实现一个 `attempt()`。oracle 与"agent 是真是假"完全解耦——假 agent 和真 agent 产出同样形状的世界状态，被同一套 oracle 打分。这意味着**真实适配器接进来时，下游的判定/报告逻辑不用改**。这是好设计。
2. **依赖极简。** 只有 zod + yaml，没有臃肿框架，攻击面小、可维护。
3. **诚实的细节，说明作者懂测试。** 比如：报错的体检项从评分里剔除（不让网络抖动伪造扣分）；满分 mock 被排除出落差（不让对照组抹平卖点）；没跑的格子显示"—"而非"FAIL"（区分"没测"和"失败"）。这些都是有经验的人才会处理的边界情况。
4. **"先无密钥接通管道"是务实的工程顺序。** 在烧任何 token 之前先证明整条流水线连通。

### 4.2 主要技术风险（按重要性排序）

1. **Oracle 覆盖率 = 产品生死线。** 如果很多任务最终都只能用 LLM 裁判（而非程序化判定），分数可信度崩塌，整个产品的价值就没了。缓解：oracle 优先、选 oracle 友好的品类（Asana）、判不了就砍掉那个任务。**这是你做商业判断时最该追问的技术问题：每进入一个新品类，先问"有多少比例的真实任务能拿到程序化判定？"**
2. **Harness 漂移。** Claude Code / Codex 几乎每周更新，更新会移动分数。一个分数离开了 `harness@版本 + 模型 + 标准集版本` 就没有意义。缓解：一切锁版本、每个结果里都记录。这同时也是护城河（维护这层很烦 = 别人难抄），是双刃剑。
3. **成本。** 每次尝试都烧真实 token。一个有 8 任务 × 多 harness × K 次重试的矩阵，跑一轮的 token 成本是真金白银（记在用户密钥上）。缓解：`max-turns` 上限、先跑便宜的 minimal、给用户花费上限。**作为持续 CI 跑的产品，单位经济学（每次跑多少钱）需要算清楚。**
4. **贡献的 target pack = 不可信代码向量。** target pack 里含**可执行的** check/setup/reset 代码，开源生态邀请社区贡献 = 邀请陌生人的代码在用户机器上跑。文档已识别这个风险，方案是容器隔离 + 网络范围限制 + （待定的）pack 审核/签名机制。但这是真实的安全债，规模化前必须解决。
5. **任务依赖/状态管理还很原始。** 比如"指派任务"依赖"先创建了任务"，但当前系统不排序、也不在前置条件没满足时干净失败——只是隐含在 prompt 里。真实运行 + 沙箱 reset 接进来后，这块要补。

### 4.3 还没决定的开放问题（影响商业判断）

文档自己列的、尚未拍板的：
- **开源/付费的精确分界线**——免费 skill 能力给到多强才不会蚕食付费层？
- **定价单位**——按次跑？按席位？按目标？还是 CI 闸门订阅？
- **谁来"加持"这个 harness 标准**——我们 + 设计伙伴，还是推社区标准？（决定 schema 采纳是不是真护城河。）
- **自测 vs 服务**——第一天就做自助产品，还是先做高接触的"代测服务"学习再产品化？
- **最小 harness 用什么模型、多简陋。**

2. 跑通后，立刻做"落差"的真实版：静态分 vs 真实 agent 通过率。这一张真实的对比图就是你的销售第一张幻灯片。

| 术语 | 一句话解释 |
|---|---|
| **AX（Agent Experience）** | Agent 体验——agent 如何发现、理解、操作你的产品。本项目认领的"伞"。 |

| **OpenAPI** | 机器可读的 REST API 描述文件（端点/参数/返回）。 |

| **BYOK** | 自带密钥，用户付自己的推理费，我们不碰密钥。 |
